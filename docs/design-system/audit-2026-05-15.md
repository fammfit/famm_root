# UI Audit Report — 2026-05-15

Read-only audit. Compares the current state of UI code in this
repository against the FAMM design system as documented in
`docs/DESIGN_SYSTEM.md`, `docs/FIGMA_VARIABLES.md`,
`docs/design-system/components/`, and `docs/design-system/patterns/`.

No code was modified to produce this report.

---

## Summary

- Files audited: every `.tsx` under `packages/ui/` and
  `apps/web/src/components/`.
- Critical issues: **5** themes.
- Moderate issues: **4** themes.
- Minor issues: **3** themes.
- Top-level finding: **the L1 design system is healthy and follows
  its own rules — but `apps/web` does not consume it.** The booking
  feature ships its own `apps/web/src/components/ui/{Button,Skeleton,Toast}.tsx`,
  uses raw Tailwind palette colors throughout, and contains 16 raw
  `<button>` and 1 raw `<input>` instead of `@famm/ui` primitives.

The fastest path to compliance is consumption, not new primitives.
The L1 / L2 layer is in better shape than the products that should
consume it.

## 2. Duplicate components (CRITICAL)

Three primitives are re-implemented at L3, bypassing `@famm/ui`:

| Path                                                  | Duplicates                       |
|-------------------------------------------------------|----------------------------------|
| `apps/web/src/components/ui/Button.tsx`               | `@famm/ui` `Button`              |
| `apps/web/src/components/ui/Skeleton.tsx`             | `@famm/ui` `Spinner` / planned `Skeleton` |
| `apps/web/src/components/ui/Toast.tsx`                | planned `@famm/ui` `Toast`       |
| `apps/web/src/components/ui/Sheet.tsx`                | planned `@famm/ui` `Modal` / `Drawer` (bottom-sheet variant) |

`apps/web/src/components/ui/Button.tsx:18` uses
`bg-gray-900 text-white` and `text-[15px]` — both blocked by the
design system rules in `CLAUDE.md`. These shims are the principal
reason no `@famm/ui` imports were detected in `apps/web` during the
audit.

**Severity: Critical.** Blocks the "one approved primitive per
concern" guarantee. Every duplicate must be removed and call sites
migrated to `@famm/ui`.

## 3. One-off styles (CRITICAL)

### 3.1 Raw color values

102 occurrences of raw Tailwind palette classes
(`gray-{200..900}`, `red-{50..800}`, `emerald-{100..600}`,
`amber-{100..800}`, `blue-{100..700}`) across the `booking`,
`chat`, and `ui` folders under `apps/web/src/components/`.

Representative samples:

- `apps/web/src/components/booking/BookingCard.tsx:13-18` — status
  color map keyed on Tailwind palette names.
- `apps/web/src/components/ui/Toast.tsx:50-52` — `bg-emerald-600`,
  `bg-red-600`, `bg-gray-900`.
- `apps/web/src/components/booking/BookingConfirmSheet.tsx:127` —
  `border-gray-200`, `focus:border-gray-900`.

None of these consume the semantic families allowed by
`CLAUDE.md` (`surface-*`, `text-*`, `accent-*`, `signal-*`,
`border-*`). All 102 are token-system bypasses and would be flagged
by `packages/config/eslint/design-system.js` once it lints `apps/web`.

### 3.2 Raw px / arbitrary values

15+ arbitrary-value classes:

- `Button.tsx:29` — `text-[15px]`.
- `Sheet.tsx:62` — `max-h-[92vh]`, `max-h-[85vh]`.
- `Toast.tsx:49` — `max-w-[92vw]`.
- `TrainerDayView.tsx:47,56,74,77,164` — `text-[11px]`,
  `text-[12px]`, `ml-[52px]`.
- `BookingCard.tsx:43,57` — `text-[10px]`.
- `DayStrip.tsx:54,65` — `h-[68px]`, `text-[11px]`.
- `MonthCalendar.tsx:80,122` — `text-[11px]`, `min-w-[16px]`.
- `SlotGrid.tsx:62,65` — `text-[13px]`, `text-[10px]`.
- `BookingsList.tsx:73-75` — `h-[88px]`.

**Severity: Critical.** These bypass the typography and spacing
scales documented in `docs/FIGMA_VARIABLES.md` §3 and §2.

### 3.3 Raw HTML primitives

16 `<button>` and 1 `<input>` in product code:

- `BookingCard.tsx:32`, `SlotGrid.tsx:51,73`, `MonthCalendar.tsx`,
  `DayStrip.tsx`, and others — `<button>` with inline focus / hover /
  active utilities instead of `Button` from `@famm/ui`.
- `chat/ChatWindow.tsx:137` — raw `<input type="text">` instead of
  `Input`.

**Severity: Critical.** Documented as banned in `CLAUDE.md`,
`button.md §5`, `input.md §4`.

### 3.4 Inline `style={}` outside L1

`TrainerDayView.tsx:40,45,67,165` and `DayStrip.tsx:38` use
`style={{}}` for dynamic calendar positioning (hour-row offsets,
scrollbar hiding). These are **acceptable** because the values are
computed at runtime and not expressible by Tailwind classes — but
they should be commented as exceptions and isolated.

**Severity: Moderate.** Document the exception or wrap in a small
utility component.

## 4. Components missing states

Per the specs in `docs/design-system/components/`:

| Component | Documented states | Implemented states | Gap |
|-----------|-------------------|---------------------|-----|
| `Button`  | default / hover / focus-visible / active / disabled / loading / error | default / hover / focus-visible / active / disabled / loading | `error` (transient flash) — doc says it exists; code does not implement. Lower-priority since the doc itself frames it as optional. |
| `Input`   | default / filled / hover / focus-visible / disabled / read-only / error / with-hint | default / focus-visible / disabled / error / with-hint | `hover` and `read-only` not visually distinguished. |
| `Badge`   | default / focus                                                       | `focus` (not `focus-visible`) | Badge uses `focus:outline-none focus:ring-2` instead of `focus-visible:*` — this fires on mouse focus too. Inconsistent with Button / Input. |
| `Card`    | default / focus-within                                                | default; no `focus-within` ring | `focus-within` not implemented — invisible cards on touch viewports when an inner control has focus. |
| `Spinner` | (no spec)                                                             | size scale only | Doc gap, not state gap. |

**Severity: Moderate.** Badge focus inconsistency is the most
visible; Card `focus-within` is the most accessibility-impacting.

## 5. Components missing accessibility features

`Button` (`packages/ui/src/components/primitives/button.tsx`):

- **No default `type="button"`.** Spec
  (`docs/design-system/components/button.md §10 #14`) says the
  default `type` is `button`. Code does not enforce — a Button
  inside a form defaults to `type="submit"`, which is the native
  HTML behavior the spec explicitly aims to prevent.
- **No runtime enforcement of icon-only `aria-label`.** Spec §6
  requires it; code is silent. Acceptable as a CI/lint check, but
  a dev-mode `console.warn` would catch it earlier.
- **Forbidden combinations not enforced.** `variant="destructive"` +
  `size="sm"` and `variant="link"` + `size="icon"` are both legal at
  the type level. A dev-mode warning would surface these.

`Badge` — `focus` instead of `focus-visible` (see §4 above).

Feature components — many `<button>` elements with no
`aria-label`, manually-rolled focus rings using raw colors
(`focus-visible:ring-gray-900` at
`apps/web/src/components/booking/BookingCard.tsx:37`,
`apps/web/src/components/ui/Button.tsx:54`), and no
reduced-motion fallback.

**Severity: Critical.** The drift here directly violates
`docs/DESIGN_SYSTEM.md §5`.

## 7. Components that should be refactored

In priority order:

1. **`apps/web/src/components/ui/Button.tsx`** — delete; replace
   call sites with `import { Button } from "@famm/ui"`. The shim
   carries raw colors, `text-[15px]`, and its own focus ring.
2. **`apps/web/src/components/ui/Toast.tsx`** — keep until the
   planned `@famm/ui Toast` ships; tighten by removing raw color
   palette usage in the interim (use semantic tokens via Tailwind
   utilities already in scope).
3. **`apps/web/src/components/ui/Skeleton.tsx`** — same: keep until
   the planned `Skeleton` primitive ships; semantic-token its
   colors meanwhile.
4. **`apps/web/src/components/ui/Sheet.tsx`** — keep until the
   planned `Modal` / `Drawer` ships. Semantic-token meanwhile.
5. **`booking/*.tsx`** — replace raw `<button>` with
   `<Button>`, replace status color maps with `Badge` `success` /
   `warning` / `danger`, replace raw text size classes with semantic
   utilities (`text-body-sm`, `text-caption` etc., once
   `tokens.json` ships them).
6. **`chat/ChatWindow.tsx`** — replace raw `<input>` with `Input`
   wrapped in a `FormField` (or just `Input` if the surface doesn't
   benefit from a label row).
7. **`Badge`** in L1 — change `focus:*` to `focus-visible:*` to
   match `Button` and `Input` and to comply with the spec.
8. **`Button`** in L1 — add default `type="button"`, dev-mode
   warnings for forbidden combinations and icon-only without
   `aria-label`.
9. **`Card`** in L1 — add the `interactive` variant documented as
   "reserved" in `card.md §2`, gain `focus-within` ring on default.

## 8. Recommended implementation order

Sequenced by leverage and dependency:

### Phase 1 — Stop the bleeding (this week)

1. **Migrate `apps/web` to `@famm/ui`.** Replace
   `import ... from "./components/ui/Button"` with
   `import { Button } from "@famm/ui"` across the booking and chat
   features. This single PR closes most of §3.3 above.
2. **Delete `apps/web/src/components/ui/Button.tsx`** once nothing
   imports it.
3. **Enable the design-system ESLint config on `apps/web`.**
   `packages/config/eslint/design-system.js` exists; once
   `apps/web` consumes it, the 102 raw color violations and the 15+
   arbitrary `text-[..]` cases will start blocking PRs.

### Phase 2 — Fix the primitives (this week)

4. **Badge** — change `focus:*` to `focus-visible:*`.
5. **Button** — default `type="button"`; dev-mode warnings.
6. **Card** — add `interactive` variant + `focus-within` ring.

### Phase 3 — Document what's shipped (next week)

7. **Write specs for `Spinner`, `StatCard`, `FormField`,
   `SessionTimer`.** Use `FIGMA_COMPONENT_TEMPLATE.md` §1 as the
   template.
8. **Add a spec for the L3 shims** (`Sheet`, `Toast`, `Skeleton`)
   noting their planned replacement, so the drift doesn't grow.

### Phase 4 — Replace the remaining shims (next month)

9. **Ship `Skeleton` and `Toast` primitives** per their specs;
   migrate `apps/web/src/components/ui/{Skeleton,Toast}.tsx` call
   sites; delete the shims.
10. **Ship `Modal` primitive**; migrate `Sheet.tsx` usages to the
    `Modal` `full` variant on mobile (which renders as a bottom
    sheet per `modal.md §6`); delete `Sheet.tsx`.

### Phase 5 — Cover the gaps (rolling)

11. **Build the planned primitives in dependency order:**
    `Select`, `Checkbox`, `Radio`, `Tabs`, `Accordion`, `Tooltip`,
    `Navigation`, `Table`. Each ships with: code + tests +
    Storybook + the docs already exist.

### What this report explicitly does NOT recommend

- Rewriting the booking flow visually. The job is to migrate it to
  primitives, not redesign it.
- Building new primitives ahead of consumption. Don't ship a
  `MultiSelect` before `Select` has users.
- Touching `tokens.json` by hand. It regenerates from Figma per
  `DESIGN_SYSTEM.md §4.2`.
- Adding a new icon library or motion library. `lucide-react` and
  the planned `useReducedMotion` hook are the sanctioned choices.
